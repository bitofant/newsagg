# Databases (`src/db/`)

Two SQLite-backed modules: `news.ts` (topics + articles) and `users.ts` (users, votes, signals, prefs, read state). Both use Node's built-in `node:sqlite`.

Column mapping snake_case (SQL) → camelCase (TS) is done at the DB layer boundary in each module.

## News DB (`src/db/news.ts`)

- **`topics`** — high-level topic list with terse descriptions (used by consolidator for matching).
  - `summary` (string): LLM-generated 2-3 sentence summary, created when topic reaches 2+ articles, regenerated on `substantial_new_info`.
  - `bullets`, `new_info` (JSON array of strings, nullable): used by long-running topics that have crossed the bullets threshold. `bullets` is the running list of material developments; `new_info` is what's materially new since the previous regeneration (rendered with a "NEW:" prefix in the UI).
  - `substantial_event_timestamps` (JSON array of unix-ms numbers, nullable, treated as `[]` when null): one timestamp appended per `substantial_new_info` event. Length doubles as the threshold counter for switching prose-only → bullets format; the values themselves are also useful for surfacing recency in the UI.
  - `embedding` (BLOB, nullable): little-endian Float32 bytes from the consolidator's pre-filter embedder. Refreshed at every `createTopic` and at the end of every `regenerateTopicSummaries`. See `@docs/embeddings.md`.
  - `embedding_model` (TEXT, nullable): the `Embedder.model` value that produced the bytes. Topics whose stored model doesn't match `config.embedding.model` are treated as missing and re-embedded by the backfill loop on next startup, so model swaps never silently produce mixed-vector cosines.
- **`articles`** — title, text, timestamp, source, url. `articles.topic_id` column remains (SQLite can't drop columns) but is vestigial.
- **`article_topics`** — junction table for the many-to-many article↔topic relationship (see decision below). All queries use this; the legacy `articles.topic_id` is unused.

## User DB (`src/db/users.ts`)

- **`users`** — email, password hash, `interval_ms` (per-user front-page schedule, see aggregator), `manual_preferences`, `preference_profile`.
  - `manual_preferences`: user-maintained free-form text — **only** the user edits this. Treated as authoritative input to the profiler.
  - `preference_profile`: LLM-generated markdown derived from `manual_preferences` + vote history; **not user-editable**, regenerated by the debounced profiler. Displayed read-only in `/settings`. This is the field the aggregator reads for relevance scoring.
- **`user_votes`** — thumbs up/down per (user, article); upsert on change.
- **`signal_queue`** — consolidator → aggregator messages: `new_topic`, `added_to_topic`, `substantial_new_info`, `concluded_issue`. See "Signal queue in SQLite" below.
- **`user_read_topics`** (`user_id, topic_id, read_at`) — tracks which topics a user has marked as read; managed atomically via `setReadTopics()` (see "Read line as atomic state replacement" below). `replaceReadTopic(oldId, newIds)` is used by topic unmerging; `unionReadTopic(loserId, winnerId)` is used by topic merging (`INSERT OR IGNORE` so an existing winner `read_at` is preserved).
- **`front_pages`** — generated front pages persisted as JSON blobs (see "Front pages persisted in SQLite" below).

## Design decisions

### `node:sqlite` over `better-sqlite3` (2026-04-07)
Node 22.5+ ships built-in SQLite. Avoids native addon compilation issues (which broke on Node 25) and removes a heavyweight dependency. The API is synchronous like `better-sqlite3`. It's marked "experimental" but is stable enough for this use case.

### Signal queue in SQLite, not in-memory (2026-04-07)
Consolidator signals are written to a `signal_queue` table rather than kept in memory. This survives process restarts and means no signals are lost if the aggregator hasn't consumed them yet.

### Front pages persisted in SQLite (2026-04-07)
Generated front pages are stored in a `front_pages` table, not an in-memory Map. Consistent with the signal-queue decision: if signals survive restarts, front pages should too. The aggregator reads from DB on demand rather than caching.

### Many-to-many article-topic relationship (2026-04-14)
Articles are linked to topics via an `article_topics` junction table, not a single `topic_id` FK. A roundup article covering US-Iran diplomacy and Artemis II should belong to both topics, not merge them. The legacy `articles.topic_id` column remains (SQLite can't drop columns) but is vestigial — all queries use the junction table. The consolidator's LLM prompt returns `topicIds: number[]` per article. Topic summaries are instructed to focus only on aspects relevant to the specific topic, since shared articles may cover unrelated subjects.

### View-gated signals → read-topic tracking (2026-04-13)
The signal `consumed` flag and `last_viewed_at` are no longer used. The aggregator reads a 14-day rolling window of all signals regardless of consumption state. Users mark topics as read via a "read line" UI divider (`POST /api/readtopics`), and read topics are excluded from future front-page generation. `user_read_topics(user_id, topic_id, read_at)` table tracks read state. Old signals are cleaned up hourly. Rationale: the view-gated model only showed signals since last visit; the rolling window + read tracking gives users up to 100 topics from the last 2 weeks and lets them progressively work through them.

### Read line as atomic state replacement (2026-04-13)
`POST /api/readtopics` receives the full set of topic IDs that should be read; the server does a transactional delete-all + re-insert for the user. Clicking a "mark above as read" divider in the middle of the list also un-reads anything below it that was previously read. Rationale: simpler than incremental add/remove operations, and the read-line metaphor naturally implies a single position in the list. (Single-topic toggles use a separate delta endpoint, `PUT /api/readtopics/:topicId`.)

### Manual + generated preference profiles as separate columns (2026-04-28)
Previously `users.preference_profile` was both auto-generated by the profiler AND user-editable in `/settings`, so any hand-edited content was silently overwritten on the next regeneration. Split into two columns: `manual_preferences` (user-only, never touched by the profiler) and `preference_profile` (LLM-only, never touched by the user). The profiler reads the manual field and feeds it into the prompt as "HARD PREFERENCES — treat as authoritative, do not contradict or dilute"; the LLM produces a unified profile that preserves the manual content verbatim and augments it with vote-inferred interests. The aggregator continues reading only `preference_profile` — no aggregator changes. PATCH `/api/preferences` strict-rejects `preferenceProfile` so callers can't accidentally write to the generated field. Manual edits trigger the same 15-min debounced regeneration as votes, so a brand-new user who writes a manual profile gets relevance scoring without having to vote first. Rejected alternatives: (a) a single field with embedded "do-not-edit" markers — fragile, the LLM can drop or paraphrase the markers; (b) one-time copy of pre-existing `preference_profile` content into `manual_preferences` on migration — would silently re-classify auto-generated text as authoritative, the wrong default. Migration leaves existing `preference_profile` values intact; they get overwritten at next regeneration.
